## 引言：CLI 的动词革命

如果你用过传统的命令行工具，你会发现它们的动词都相当"正经"：install、remove、start、stop、build、deploy。这些动词符合人们对机器的直觉——你要让机器做什么，就用对应的动作词。

但 Claude Code 不太一样。它用的是：**compact、resume、doctor、undo、diff**。

这些动词看起来像是在描述某种**状态**或**关系**，而不只是在下达指令。这就是我所说的"奇怪动词"——它们不是传统的"做 X"命令，而是更像在描述一种**对话状态**或**交互模式**。

## 会话控制：把 AI 当作活的对话

### `/compact` — 最奇怪的动词

如果只能选一个最独特的命令，那一定是 `/compact`。在 Claude Code 之前，你见过哪个 CLI 工具需要"压缩"自己的？

```bash
/compact                    # 自动压缩
/compact 保留这个函数定义   # 自定义压缩
```

这个命令的设计哲学很有意思：它承认了 AI 对话有一个根本问题——**上下文会越来越长，越来越贵**。传统工具选择无视这个问题，或者让你手动清理。但 Claude Code 选择让你主动"整理"对话，而且还能精确控制保留什么。

这就像一个图书管理员帮你整理书架，不是简单地把书扔掉，而是问你："这些书还要留着吗？"

### `/resume` — 不是 continue

有趣的是，Claude Code 用的是 `resume`（恢复）而不是 `continue`（继续）。

区别在哪？
- **continue** 暗示对话从未停止
- **resume** 暗示对话曾经中断，现在要接上

这个细微的差别体现了 Claude Code 对**会话生命周期**的理解：AI 对话不是线性的，而是可以中断、可以恢复的。

### `/undo` — 撤销的艺术

`/undo` 是另一个有意思的设计。它不是简单的"撤销上一次操作"，而是能理解操作的上下文。

在传统版本控制中，undo 通常意味着"回到上一个状态"。但 Claude Code 的 undo 更智能——它理解代码的逻辑结构，能撤销有意义的变更，而不是机械地按步骤回退。

## 模式切换：动词即状态

### `/plan` — 不是计划模式，是计划模式

`/plan` 这个命令很有趣。当你在代码库里输入 `/plan` 时，Claude Code 会切换到只读模式——它可以分析、搜索、给你建议，但不会修改任何文件。

这本身不奇怪，奇怪的是这个设计揭示了一个洞察：**分析和执行是两个完全不同的心智模型**。

传统 CLI 通常只有一种模式：你要么在命令行里敲命令，要么不在。但 Claude Code 认识到，当你需要"规划"时，你的需求和"执行"完全不同。`/plan` 不是一个命令，而是一个**状态切换**。

类似的还有：
- `/fast` — 切换到快速响应模式
- `/think` — 切换到深度思考模式
- `/vim` — 切换 vim 键绑定

这些都是**状态**，不是动作。

## 诊断工具：把 AI 当作病人

### `/doctor` — 最有人情味的命令

`/doctor` 是 Claude Code 最"奇怪"也最可爱的命令之一。

想象一下：一个 AI 工具，让你输入 `/doctor` 来诊断自己的问题。这在软件工程历史上几乎是前所未有的。

传统工具会怎么做？给你一个 error log？让你去 Stack Overflow 搜？Claude Code 直接内置了一个"医生"——它会检查：
- 网络连接
- 权限配置
- MCP 服务器状态
- 项目配置

这反映了一种**自省**的设计哲学：一个工具如果不能诊断自己的问题，怎么能帮你诊断代码的问题？

### `/context` — 让你知道自己在哪

`/context` 会显示当前对话使用的数据来源。这看起来是个小功能，但它的存在揭示了一个重要的认知：**AI 的输出不是凭空来的，它有自己的"上下文"**。

通过 `/context`，用户可以看到：
- 哪些文件被读取了
- 哪些文档被引用了
- 上下文的范围有多大

这是透明度的体现，也是建立信任的方式。

## 技能命令：动词的延伸

### `/commit` — 智能提交

`/commit` 不是简单的 `git commit`。它会：
1. 分析你所有的代码变更
2. 理解改动的目的和影响
3. 生成符合规范的提交信息

这反映了 Claude Code 的一个核心理念：**动词应该理解意图，而不是执行机械操作**。

### `/review-pr` — PR 审查

`/review-pr` 把 AI 能力直接集成到代码审查流程中。你不需要复制 PR 链接到其他地方，Claude Code 直接就能获取 PR 详情并分析。

## 这些"奇怪"背后的设计哲学

回顾这些命令，我发现几个共同点：

### 1. 动词即状态

传统 CLI 的动词是瞬时的：`git commit`、`npm install`。动作执行完，动词的意义就消失了。

但 Claude Code 的动词更多是在描述一种**状态**或**模式**。`/plan` 不是在"计划"，而是进入"计划模式"；`/compact` 不是在"压缩"，而是在"整理对话"。

### 2. 承认 AI 的特殊性

`/compact`、`/context`、`/cost` 这些命令都承认了一个事实：AI 对话和传统命令行有本质区别。对话会增长、上下文会累积、成本会变化。传统工具不关心这些，因为它们处理的是静态数据；CLI 工具关心这些，因为 AI 对话是动态的。

### 3. 自省能力

`/doctor`、`/status`、`/context` 都体现了自省能力。工具能理解自己的状态，这让它更像一个合作伙伴，而不是一个死板的执行器。

### 4. 意图导向

`/commit` 不是执行 git 命令，而是理解你想提交什么；`/resume` 不是继续对话，而是恢复对话的上下文。这些命令不是在做"动作"，而是在理解"意图"。

## 结语

Claude Code 的"奇怪动词"不是故作独特，而是一套完整的设计哲学的体现：把 AI 对话当作一个需要管理、诊断、维护的生命体，而不是一个简单的命令执行器。

下次当你输入 `/compact` 时，不妨想想：这是一个革命性的交互范式——我们不再是在"使用工具"，而是在和工具"对话"。
